## Слабые места и риски

### 1. ZAP: формат `zap-baseline-rules.tsv` (исправлено в коде)

В логе:

`Failed to load config file /zap/wrk/rules.tsv Unexpected number of tokens on line - there should be at least 3, tab separated: 10027 IGNORE`

**Причина:** OWASP ZAP baseline ожидает **не меньше трёх полей, разделённых табуляцией** (Rule ID, Action, Fingerprint). Строка `10027	IGNORE` без третьей колонки **невалидна**.

**Сделано:** в [`test/config/zap-baseline-rules.tsv`](config/zap-baseline-rules.tsv) строка приведена к виду `10027	IGNORE	*` (или эквивалент с явной третьей колонкой).

**Риск до фикса:** правила игнорирования не применялись; скан шёл с дефолтным набором, предупреждения в логах шумят.

---

### 2. Docker: монтирование `rules.tsv` (первый запуск ZAP)

В логе для одного из запусков:

`error mounting ".../zap-baseline-rules.tsv" to rootfs at "/zap/wrk/rules.tsv": ... mountpoint ... is outside of rootfs`

**Характер:** типично для **Docker Desktop + virtiofs** при монтировании **одиночного файла** в путь внутри контейнера.

**Рекомендации:** монтировать **каталог** `test/config` в `/zap/wrk/config` и вызывать `zap-baseline.py -c config/zap-baseline-rules.tsv`, либо копировать файл в `test/reports/` перед запуском (каталог уже смонтирован как `/zap/wrk`). При необходимости можно доработать [`test/scripts/run-zap-baseline.sh`](scripts/run-zap-baseline.sh).

---

### 3. Повторный прогон миграций на существующей БД

В логе при `migrate`:

- `0013`: ошибка по колонке `client_secret` (уже удалена миграцией `0017` на схеме, где порядок/состояние отличается от «чистой» установки).
- `0016`: сообщения о уже существующих FK.

**Вывод:** цикл `for f in migrations/*.sql | psql` на **непустой** БД даёт шум и частичные ошибки. Для продакшена обычно используют **один проход миграций** (sqlx migrate / flyway / отдельная версионность), а не повторный залив всех файлов.

**Рекомендация:** для CI/local — `docker compose down -v` перед полным регрессом; для прод — только инкрементальные миграции с версией.

---

### 4. Semgrep: одно срабатывание вне гейта ERROR

Лог: `1 finding (1 blocking)` при этом `OK: Semgrep no ERROR findings`.

**Интерпретация:** под **blocking** в UI Semgrep может пониматься не severity ERROR; **политика репозитория** — падать только на **ERROR**. Имеет смысл один раз открыть SARIF/вывод и **классифицировать** finding (false positive / понизить правило / исправить код).

---

### 5. Операционные напоминания (не регресс кода)

- **OIDC discovery:** пустой `OIDC__SERVER_METADATA_URL` — встроенный документ; непустой URL должен быть **доступен** из контейнера backend, иначе 502 (или включён controlled fallback `OIDC__METADATA_PROXY_FALLBACK`).
- **HSTS:** на локальном HTTP ожидаемо отсутствует — в прод выставлять на **TLS-терминаторе**.
- **Сид для escalation** (`security-escalation-probe@...`) — только для тестового контура; в общей БД не смешивать с реальными пользователями без контроля.

## Готовность «всё готово»?

| Критерий | Готовность |
|----------|------------|
| Зависимости (npm / Rust advisories) | Да |
| Образы (Trivy по политике) | Да, при актуальном `.trivyignore`/обновлении баз |
| HTTP hardening (backend + nginx CSP) | Да, по последнему прогону headers |
| RBAC negative probe | Да |
| Чистота логов CI / повторяемость ZAP + миграций | **Частично** — см. п. 1–3 выше |

**Итог:** ядро и периметр HTTP для описанного прогона выглядят **готовыми к выкатке с оговорками**; до «без сюрпризов в CI» стоит **стабилизировать ZAP mount** (при необходимости) и **не полагаться на повторный полный залив SQL** на долгоживущей БД.
